# 基于大数据计算引擎的在线判题系统的设计与实现

## 中文摘要
随着程序设计教育、算法训练与在线考试场景的快速发展，在线判题系统已经成为高校实践教学、编程能力测评和算法竞赛组织中的重要基础平台。传统在线判题系统通常将关注重点放在题目管理、代码提交、编译执行与结果返回等核心业务流程上，能够满足基础的评测需求，但在判题数据的长期沉淀、用户行为分析、题目质量评估以及系统运行趋势预测等方面仍然存在明显不足。与此同时，随着系统运行时间的增长，平台会积累大量提交记录、判题结果、运行时间、内存占用和用户行为日志，这些数据不仅能够反映学生编程能力和题目难易程度，还能够为平台运维、题库建设和教学决策提供重要依据。因此，将大数据计算引擎引入在线判题系统，构建集判题执行、离线分析、预测评估和可视化展示于一体的一体化平台，具有较高的工程价值与研究意义。

本文围绕“基于大数据计算引擎的在线判题系统的设计与实现”这一主题，设计并实现了一套面向教学与训练场景的在线判题系统。系统后端采用 Spring Boot 框架构建单体应用，负责用户认证、题目管理、测试用例管理、代码提交、判题任务调度和结果查询等业务功能；主数据库采用 MySQL，完成业务数据与分析结果数据的持久化存储；判题执行层采用 Docker 容器沙箱机制，针对 C、C++、Java、Python 等语言提供隔离的编译与运行环境，通过网络禁用、CPU 与内存限制、只读挂载和进程数控制等方式提升判题安全性；前端采用 React、TypeScript、Ant Design 和 ECharts 构建可视化页面，实现普通用户与管理员的分角色访问。

在大数据分析方面，本文构建了“业务数据库—HDFS—Hive/Spark—MySQL—前端页面”的离线分析链路。系统在每次提交判题完成后，将关键指标写入分析事件表，再由应用定时或手动导出至 HDFS；Hive 用于完成按天的基础统计和描述性分析，包括全站提交趋势、题目维度统计、用户维度统计、语言分布、结果分布、活跃度和高峰时段等内容；Spark 在已有 Hive 数据基础上进一步实现预测与评估能力，包括未来提交量预测和基于历史判题行为的题目难度评估与标签生成。通过将离线计算结果回写至 MySQL，系统实现了分析结果的产品化落地，使管理员能够在后台页面中直观查看系统运行状况、题目质量与预测结果，普通用户也能够查看自身提交趋势、语言偏好和结果分布。

本文从需求分析、总体设计、关键功能实现、数据分析链路构建、预测模块实现以及系统测试等方面展开论述，并通过功能测试、判题场景测试、数据导出测试、Hive 聚合测试、Spark 预测测试和前端展示验证，证明系统具备较好的完整性、稳定性与可扩展性。研究结果表明，将 Hadoop、Hive 与 Spark 等大数据计算组件引入在线判题系统，能够有效提升平台对历史数据的利用深度，为教学分析、题目运营和系统优化提供支撑。本文所设计与实现的系统为在线判题平台与大数据分析能力的融合提供了一种可行方案。

**关键词**：在线判题系统；Spring Boot；Docker；Hadoop；Hive；Spark；数据分析；难度评估

## Abstract
With the rapid development of programming education, algorithm training, and online examination scenarios, online judge systems have become an important infrastructure in universities for practical teaching, programming assessment, and algorithm competitions. Traditional online judge systems mainly focus on core business functions such as problem management, code submission, compilation, execution, and result feedback. Although these functions can satisfy basic evaluation requirements, such systems are often weak in long-term data accumulation, user behavior analysis, problem quality assessment, and operational trend prediction. As the system runs continuously, a large amount of submission records, judging results, execution time, memory consumption, and user behavior logs are generated. These data can not only reflect students' programming ability and problem difficulty, but also provide valuable support for platform maintenance, problem set construction, and teaching decisions. Therefore, introducing big data computing engines into an online judge system and building an integrated platform that combines judging execution, offline analysis, predictive evaluation, and visualization has both practical engineering value and research significance.

This thesis focuses on the design and implementation of an online judge system based on big data computing engines. A complete system for teaching and training scenarios is designed and implemented. The backend adopts Spring Boot to build a monolithic application responsible for user authentication, problem management, testcase management, code submission, judging task scheduling, and result query. MySQL is used as the primary database to persist both business data and analysis results. Docker containers are introduced as the judging sandbox. For languages such as C, C++, Java, and Python, isolated compilation and runtime environments are provided. Security is enhanced through network isolation, CPU and memory limits, read-only mounts, and process count control. The frontend is built with React, TypeScript, Ant Design, and ECharts to provide role-based visualization pages for ordinary users and administrators.

For big data analytics, this thesis constructs an offline data pipeline of “business database - HDFS - Hive/Spark - MySQL - frontend pages”. After each judging task is completed, key indicators are written into an analysis event table and then exported to HDFS either manually or on schedule. Hive is used to perform daily basic aggregation and descriptive analysis, including platform-wide submission trends, problem-level statistics, user-level statistics, language distribution, verdict distribution, active user analysis, and peak-hour analysis. On top of the Hive data, Spark is further introduced to implement predictive and evaluative capabilities, including future submission volume forecasting and problem difficulty assessment with automatic difficulty labeling based on historical judging behavior. By writing offline results back to MySQL, the system realizes productized delivery of analytical results, enabling administrators to directly observe platform status, problem quality, and prediction outcomes on dashboard pages, while ordinary users can view their own submission trends, language preferences, and verdict distributions.

This thesis discusses the system from the aspects of requirement analysis, overall design, key function implementation, data analysis pipeline construction, predictive module implementation, and system testing. Through functional tests, judging scenario tests, data export validation, Hive aggregation verification, Spark prediction verification, and frontend display validation, the system is shown to have good integrity, stability, and scalability. The results demonstrate that integrating Hadoop, Hive, and Spark into an online judge system can significantly improve the depth of historical data utilization and provide solid support for teaching analysis, problem operation, and system optimization. The system designed and implemented in this thesis provides a feasible solution for the integration of online judge platforms and big data analytics.

**Key Words**: online judge system; Spring Boot; Docker; Hadoop; Hive; Spark; data analysis; difficulty assessment

# 第一章 绪论

## 1.1 研究背景
随着计算机科学教育的普及和信息化教学模式的发展，程序设计课程已经成为高校计算机类专业的重要基础课程。与传统理论课程相比，程序设计课程更加依赖实验、练习和持续反馈。学生只有通过不断编写代码、运行程序和修正错误，才能逐步掌握数据结构、算法设计、程序调试和复杂问题求解能力。在这一背景下，在线判题系统因其能够提供自动化代码评测、即时反馈和题库管理能力，已经被广泛应用于教学实验平台、课程作业系统、校内选拔赛和算法竞赛训练平台之中。

在线判题系统的核心任务是对用户提交的程序进行编译、执行和输出校验，并根据题目测试数据给出接受、错误答案、运行错误、超时、内存超限等结果。一个设计良好的在线判题系统能够显著提高教师批改效率，减轻人工检查负担，并为学生提供及时反馈。然而，随着系统规模扩大和运行周期增加，单纯实现“提交—判题—返回结果”这一业务闭环已经难以满足教学与管理的综合需求。平台运营者和教师希望进一步了解学生整体提交趋势、不同题目的难度差异、常见错误类型、各语言使用情况以及系统在不同时间段的负载分布，以便优化教学安排、改进题库设计和提升平台稳定性。

另一方面，近年来 Hadoop、Hive、Spark 等大数据计算框架已经在海量数据存储、批处理分析和预测建模领域得到了广泛应用。在线判题系统在长期运行过程中会积累大量结构化日志，包括提交编号、用户编号、题目编号、编程语言、判题结果、运行时间、内存占用以及提交时间等信息。这些数据具有明显的时序性、行为性和多维度特征，非常适合借助大数据计算框架进行离线统计与深度挖掘。如果能够将大数据计算引擎与在线判题系统有机结合，就可以实现从基础业务系统到数据驱动平台的能力提升，使系统不仅能够“完成判题”，还能够“理解判题数据”。

此外，随着容器技术的成熟，基于 Docker 的代码沙箱已经成为在线判题系统实现安全隔离的重要手段。相比传统直接在宿主机上执行用户代码的方式，Docker 能够提供更加清晰的环境隔离、资源限制和安全控制机制，有助于降低恶意代码、无限循环、异常进程扩散等问题对宿主机造成的风险。因此，将 Docker 判题、多语言支持、Hadoop 离线分析和 Spark 预测模块综合起来，形成一套完整的在线判题平台，既具有工程实现价值，也具备作为本科毕业设计研究对象的现实意义。

## 1.2 研究目的与意义
本文的研究目标是设计并实现一套基于大数据计算引擎的在线判题系统，在实现常规在线判题功能的基础上，进一步构建日志导出、离线统计、预测分析和可视化展示能力，从而形成集“业务执行、数据沉淀、统计分析、预测支撑”于一体的综合平台。

从教学角度看，在线判题系统能够帮助教师快速了解学生对课程知识点的掌握程度。通过统计不同题目的提交数量、通过率、错误分布和平均尝试次数，可以识别出学生普遍存在的问题，辅助教师调整教学内容和实验安排。从学生角度看，通过分析个人提交趋势、语言偏好、结果分布和最近提交记录，可以帮助学生更好地理解自身学习状态和薄弱环节，形成数据驱动的学习反馈。

从平台运营角度看，判题日志不仅反映用户行为，还反映系统负载情况和题库质量。通过对系统提交趋势、小时活跃度、语言分布和题目难度进行分析，可以更合理地安排资源、优化题库、定位异常题目，并基于历史数据构建未来趋势预测和题目难度评估模型。因此，本课题既具有工程应用意义，也体现了大数据计算技术在教育平台场景中的综合应用价值。

## 1.3 国内外研究现状
国外较早出现的在线判题平台主要服务于算法竞赛和程序设计训练，例如 UVa Online Judge、Codeforces、AtCoder、LeetCode 等。这些平台经过长期发展，在题库规模、评测性能、语言支持和用户体验方面较为成熟。其中，面向竞赛的平台更强调判题性能和高并发处理能力，面向刷题与面试的平台则更加重视题库组织、用户画像和学习路径设计。国外平台通常已经具备较强的数据分析能力，但相关实现细节大多没有公开，更多体现在推荐系统、题目难度标签和用户成长反馈等产品功能层面。

国内在线判题平台的发展也较快，既包括面向竞赛训练的系统，也包括高校课程实验平台。很多高校和培训机构基于开源系统进行二次开发，实现题目管理、代码提交和自动判题等功能。一些开源 OJ 系统如 HUSTOJ、QDUOJ、Judger 等，已经提供了较为完整的后端能力和判题逻辑，但在大数据分析与预测模块方面仍然相对薄弱，更多聚焦于业务本身。此外，部分在线教育平台虽然也具备一定的数据看板，但往往缺乏与大数据计算引擎深度结合的可复用架构。

从相关研究来看，现有工作更多集中在以下几个方面：一是安全判题机制，例如多语言隔离执行、资源限制和恶意代码防护；二是系统性能优化，例如任务队列、分布式判题和并发调度；三是程序评测算法与输出比对策略；四是学习行为分析和用户画像构建。相比之下，将 Docker 判题、Hadoop/Hive 离线分析和 Spark 预测模块整合到一个统一的在线判题系统中，并形成完整的数据产品链路的研究与工程实现仍然具有一定空间。

## 1.4 本文研究内容
本文围绕在线判题系统的实际需求，完成了以下几个方面的设计与实现工作。

第二，采用 Docker 容器构建代码评测沙箱，支持 C、C++、Java、Python 等常见语言的编译与运行，并通过 CPU、内存、进程数和网络访问控制，实现较为安全的判题环境。

第三，构建判题事件数据链路。在每次代码评测完成后，系统将判题核心信息写入分析事件表，再导出到 HDFS，借助 Hive 实现基础聚合与描述性分析，包括全站提交统计、题目统计、用户统计、语言分布、结果分布、小时活跃度等内容。

第四，引入 Spark 作为预测与评估引擎。在已有 Hive 数据基础上，首先实现未来每日提交量预测，然后进一步实现基于历史判题行为的题目难度评估与标签生成，用以支撑管理员对题库质量和系统运行趋势的观察。

第五，基于前后端分离架构完成数据可视化页面建设，为普通用户提供个人提交趋势和行为统计，为管理员提供系统分析页、预测页以及题目难度评估页，实现分析结果的产品化呈现。

# 第二章 相关技术介绍

## 2.1 Spring Boot 框架
Spring Boot 是基于 Spring 生态体系构建的快速开发框架，具有自动配置、依赖管理和内嵌 Web 容器等特点，能够显著降低 Java Web 应用的开发与部署复杂度。在本系统中，Spring Boot 被用于构建整个业务后端，包括用户管理、题目管理、提交处理、判题任务调度、分析结果查询以及认证授权等功能模块。

选择 Spring Boot 的主要原因有三点。其一，Spring Boot 的分层结构清晰，易于组织 Controller、Service、Repository 和 Entity 等模块，适合构建功能相对完整的单体系统。其二，Spring Data JPA 提供了方便的数据访问方式，能够快速实现对 MySQL 中业务表和分析结果表的增删改查。其三，Spring Security 与 JWT 机制集成方便，能够较好地支持普通用户和管理员的角色区分。

在系统实现中，Controller 层负责接收前端请求并进行参数校验；Service 层负责业务逻辑处理，如提交生成、判题执行、分析结果封装；Repository 层依靠 JPA 或原生查询访问数据库；实体类负责映射 MySQL 表结构。这样的架构层次便于维护和扩展，也为后续新增分析 API 和预测 API 提供了良好基础。

## 2.2 MySQL 数据库
MySQL 是一种广泛应用的关系型数据库管理系统，具有部署简单、使用成熟、事务支持良好等特点。在本系统中，MySQL 既承担业务数据存储，也承担分析结果回写后的查询支撑。业务数据包括用户、题目、测试用例、提交、判题任务、判题结果和分析事件；分析结果数据包括按天汇总的统计表、描述性分析结果表、提交量预测表以及题目难度评估表。

将 MySQL 作为业务数据库具有明显优势。首先，业务数据天然是结构化的，适合用关系模型建模。其次，MySQL 与 Spring Boot 集成简单，事务管理完善。再次，离线分析结果回写到 MySQL 后，前端页面无需直接连接 Hive 或 Spark，只需通过后端 API 读取 MySQL 表即可，既降低前端复杂度，也减少大数据集群对在线查询的影响。

## 2.3 Docker 容器技术
在线判题系统需要执行用户提交的代码，而用户代码具有不可控性，可能包含恶意指令、无限循环、异常进程创建、资源占用失控等风险。因此，构建安全的代码执行沙箱是在线判题系统的关键问题之一。Docker 通过容器机制提供了进程级隔离能力，使得应用能够在相对独立的运行环境中执行，同时共享宿主机内核，具备较高的性能与易用性。

本系统使用 Docker 为不同语言构建独立镜像，例如 GCC 镜像、Java 镜像和 Python 镜像。每次判题时，系统将提交代码和测试数据挂载到容器内部工作目录，在容器内完成编译和运行操作，并通过禁用网络、限制 CPU、限制内存、限制进程数、只读根文件系统和临时目录挂载等方式控制执行环境。这样既保证了评测逻辑的一致性，又降低了用户代码对宿主机造成风险的可能性。

## 2.4 Hadoop HDFS
HDFS 是 Hadoop 生态中的分布式文件系统，用于存储海量数据，具有高吞吐和面向批处理分析的特点。本系统使用 HDFS 存储判题分析事件的离线文件。每次将 MySQL 中的分析事件导出后，系统会按照日期分区写入 HDFS 目录，例如 `/oj/analysis/dt=2026-04-01`。这种按天分区的方式有利于后续 Hive 和 Spark 按日期读取与计算。

引入 HDFS 的原因在于，判题日志属于典型的可累计历史数据，适合长期沉淀并供离线分析使用。若直接在业务数据库中长时间堆积大量明细日志，会影响在线查询和业务事务性能，而 HDFS 更适合承担历史原始数据存储职责。

## 2.5 Hive 数据仓库技术
Hive 是构建在 Hadoop 之上的数据仓库工具，能够通过类 SQL 的方式对 HDFS 上的数据进行查询和聚合，适合进行离线统计分析。本系统使用 Hive 完成基础日报统计和描述性分析任务，例如每日提交总量、按题目统计、按用户统计、语言分布、判题结果分布和小时活跃度分析。

系统通过创建外部表 `oj.analysis_events` 将 HDFS 中的判题日志目录映射为结构化表，再基于 Hive SQL 对数据进行聚合，并将结果写入 HDFS 的聚合目录，最后导回 MySQL。采用 Hive 的好处在于统计逻辑清晰，SQL 易读，维护成本较低，适合实现周期性、批量性的汇总分析任务。

## 2.6 Spark 大数据计算引擎
Spark 是一种通用的大数据计算框架，具有内存计算能力强、任务执行速度快、适合批处理和机器学习扩展的特点。本系统在已有 Hive 分析基础上，引入 Spark 承担更复杂的计算任务，主要包括提交量预测和题目难度评估。

相比 Hive 更擅长 SQL 聚合，Spark 在以下两方面更适合本系统：一是基于 DataFrame 和 SQL API 可以方便地对数据进行清洗、特征构造和复杂指标计算；二是 Spark MLlib 或自定义规则模型能够支持预测与评估模块的实现，使系统在传统统计之外具备趋势预测和难度评估能力。

## 2.7 React 前端技术栈
前端部分采用 React + TypeScript + Vite 的组合，使用 Ant Design 构建表单、表格和布局组件，使用 ECharts 完成统计图表展示。前端主要分为普通用户功能页和管理员功能页两类，包含题目列表、题目详情、提交结果页、用户分析页、管理员题目管理页、系统分析页、预测页和题目难度评估页。

之所以选择 React 技术栈，是因为其组件化开发方式便于页面拆分与复用，TypeScript 有助于统一接口类型约束，Ant Design 能够快速搭建后台风格界面，ECharts 则便于绘制折线图、柱状图、饼图和趋势图，从而直观展示离线分析和预测结果。

## 2.8 本章小结
本章介绍了系统建设所依赖的核心技术，包括 Spring Boot、MySQL、Docker、HDFS、Hive、Spark 以及 React 前端技术栈。上述技术在本系统中分别承担业务实现、安全判题、数据存储、离线计算、预测分析和可视化展示等职责，共同构成了“在线判题 + 大数据分析”的技术基础。

# 第三章 系统需求分析

## 3.1 系统目标分析
本系统的总体目标是构建一套面向教学与训练场景的在线判题平台，在实现常规题目管理和自动判题的基础上，进一步引入大数据分析和预测能力，使平台不仅能够评测代码，还能够沉淀并利用判题数据，为教师、管理员和学生提供更丰富的信息支持。

从用户视角来看，系统需要为普通用户提供注册、登录、查看题目、提交代码、查看判题结果以及查看个人分析数据等功能；为管理员提供题目管理、测试用例管理、导出判题日志、查看全站分析结果和预测结果等功能。

从系统视角来看，平台需要具备安全、稳定和可扩展特性。判题部分必须隔离执行环境，避免用户代码对宿主机造成影响；数据分析部分必须能够处理不断累积的历史数据，并以较低成本生成统计结果；预测模块则需要建立在已有数据沉淀基础之上，输出对管理决策有价值的信息。

## 3.2 功能需求分析
### 3.2.1 用户管理需求
系统应支持用户注册与登录。用户注册时需要提交用户名和密码；登录后应获取令牌，用于访问受保护接口。系统应区分普通用户和管理员权限，普通用户主要用于题目浏览、代码提交与个人分析查看，管理员则额外具备题目管理、测试用例管理、分析导出与后台统计查看权限。

### 3.2.2 题目管理需求
管理员应能够新增题目、修改题目标题与描述、配置时间限制和内存限制，并管理题目对应的测试用例。测试用例管理包括新增、覆盖和删除。普通用户能够查看题目基本信息，但不能查看测试用例内容。

### 3.2.3 代码提交与判题需求
普通用户在选择题目后，应能够提交代码并指定编程语言。系统收到提交后，需要异步创建判题任务，将用户代码在 Docker 容器中完成编译与运行，并与标准输出进行比较，返回 AC、WA、RE、CE、TLE 等判题结果。用户能够查询提交状态与判题详情。

### 3.2.4 日志分析需求
系统应在每次判题完成后记录分析事件，形成包含用户、题目、语言、判题结果、运行时间、内存使用和时间戳等信息的日志数据。管理员应能够触发日志导出到 HDFS，并通过离线计算获得日报统计和专题分析结果，包括系统整体趋势、题目分析、用户分析、语言分布、结果分布和小时活跃度等。

### 3.2.5 预测与评估需求
系统除了提供描述性分析，还应具备一定的预测和评估能力。管理员应能够查看未来若干天提交量预测结果，用于观察平台负载趋势；同时应能够查看基于历史判题行为计算出的题目难度分数与难度标签，从而辅助题库管理和教学安排。

## 3.3 非功能需求分析
### 3.3.1 安全性需求
系统的安全性主要体现在两个方面：一是用户认证与权限控制，保证普通用户与管理员的功能边界清晰；二是判题执行安全，通过容器隔离、网络禁用、资源限制和目录权限控制，避免恶意代码影响宿主环境。

### 3.3.2 可靠性需求
判题任务属于系统核心业务，系统应保证提交后任务能够被稳定拉取和执行。对于运行中出现的异常，应具备重试能力和失败状态记录，避免任务长时间悬挂。同时，判题产生的工作目录应及时清理，以防止临时文件堆积。

### 3.3.3 可扩展性需求
题目类型、支持语言、分析指标和预测模型未来都可能增加。因此系统架构应尽量保持模块边界清晰，便于后续扩展更多语言镜像、增加新的 Hive 统计脚本或接入更复杂的 Spark 模型。

### 3.3.4 可维护性需求
由于本系统同时涉及后端业务、Docker 沙箱、Hadoop/Hive、Spark 和前端展示，必须通过清晰的目录结构、分层设计和相对独立的脚本体系降低维护难度。例如业务 API 与离线分析脚本解耦、分析结果通过 MySQL 回写统一提供给前端读取等，都是为了提高系统可维护性。

## 3.4 用例分析
系统主要涉及两类角色：普通用户和管理员。普通用户的主要用例包括注册、登录、查看题目、提交代码、查看提交详情、查看个人分析；管理员的主要用例包括登录、创建题目、编辑题目、管理测试用例、触发导出、查看系统统计、查看预测结果和查看题目难度分析。系统作为执行主体，还需要完成判题任务调度、日志导出、Hive 聚合和 Spark 预测。

# 第四章 系统总体设计

## 4.1 系统总体架构设计
系统采用前后端分离和单体后端相结合的总体架构。前端由 React 应用构成，负责页面交互、路由控制和数据可视化展示；后端采用 Spring Boot 单体服务，对外提供用户、题目、提交、结果、分析和预测等 REST API；MySQL 作为业务数据和分析结果数据的统一存储层；Docker 作为在线判题执行沙箱；HDFS、Hive 与 Spark 构成离线分析与预测引擎；Hadoop YARN 负责 Spark 作业资源调度。

这种架构的优点在于分工明确。在线业务请求直接访问 Spring Boot 服务，保证交互及时性；判题采用异步任务方式执行，避免长时间阻塞接口请求；分析与预测通过离线链路运行，不干扰在线事务；前端则通过统一的 API 接口从后端读取处理后的结果，避免直接依赖大数据环境。

## 4.2 系统模块划分
从功能实现角度，可以将本系统划分为以下几个核心模块。

第一，用户与权限模块。负责用户注册、登录、JWT 认证、管理员与普通用户角色区分，以及用户个人信息修改等功能。

第二，题目与测试用例模块。负责题目的创建、查询、修改以及测试用例内容录入、覆盖和删除。

第三，提交与判题模块。负责接收代码提交、创建提交记录、生成判题任务、异步拉取任务、调用 Docker 容器编译执行、输出比对以及保存判题结果。

第四，分析事件模块。负责在判题完成后记录分析事件，并支持将事件导出到 HDFS。

第五，Hive 描述性分析模块。负责按天聚合历史事件，生成全站汇总、题目维度、用户维度以及专题分析结果。

第六，Spark 预测与评估模块。负责未来提交量预测和题目难度评估，将结果写回 MySQL。

第七，前端展示模块。负责普通用户的题目访问、提交与个人分析展示，以及管理员的数据分析页、预测页和题目难度页。

## 4.3 判题流程设计
用户提交代码后，后端首先在 `submissions` 表中创建一条提交记录，并在 `judge_tasks` 表中创建对应任务。`JudgeWorker` 通过定时调度方式持续轮询待处理任务，并将任务状态从 `PENDING` 更新为 `RUNNING`。随后，`DockerJudgeRunner` 根据语言选择相应容器镜像，在容器中完成编译和运行。系统逐个读取测试用例输入文件，比较程序实际输出与标准输出，最终生成统一的 `JudgeRunResult` 对象，并将状态、耗时、内存和错误信息写入 `judge_results`。

如果判题成功结束，则提交状态更新为 `DONE`，判题结论更新为具体 verdict；如果执行过程中发生异常或超过重试次数，则任务状态会被标记为 `FAILED`。判题完成后，系统同时向 `analysis_events` 表写入一条分析事件，用于后续离线计算。

## 4.4 Docker 沙箱设计
系统为每种编程语言准备专用镜像，并在判题时通过 `docker run` 启动短生命周期容器。容器执行时采用以下安全策略：关闭网络访问、设置 CPU 和内存上限、限制进程数量、挂载只读测试数据目录、挂载可写工作目录、启用只读根文件系统以及单独的临时目录。这些策略的目标是保证容器内的用户代码无法访问外部网络、无法无限制消耗主机资源，也难以对宿主环境造成破坏。

测试数据目录采用只读挂载，避免用户代码篡改标准输入输出文件；工作目录用于保存源码文件、编译产物和程序输出文件，评测结束后及时删除；标准输出比较采用归一化策略，忽略换行风格差异和末尾空行，从而减少非本质格式问题导致的误判。

## 4.5 数据分析链路设计
系统的数据分析链路采用“在线写入、离线导出、批量聚合、结果回写”的思路。判题结束后，系统先将分析事件写入 MySQL 的 `analysis_events` 业务表。随后，管理员可以手动调用导出接口，或通过定时任务将尚未导出的事件按天写入 HDFS 对应分区目录。Hive 外表 `oj.analysis_events` 用于映射这些 HDFS 文件，进而执行基础聚合和专题分析。

聚合结果先写到 HDFS 的聚合目录，再通过脚本导入 MySQL 分析结果表。这样设计的好处在于：在线业务数据库只保存必要业务和中间日志，大规模历史统计与计算交由 Hive/Spark 完成，最终回写 MySQL 的只是较轻量的结果表，方便后端 API 快速查询。

## 4.6 预测与评估模块设计
系统当前实现了两类基于 Spark 的离线能力：其一是未来每日提交量预测，其二是题目难度评估。提交量预测使用按天聚合的历史提交数作为输入，通过简单线性回归模型估计未来七天的提交量，并将结果写入预测表；题目难度评估则根据某天各题目的 AC 率、平均尝试次数、平均耗时、运行错误比例和超时比例等指标构造难度分数，进而映射为 Easy、Medium、Hard 三个标签。

预测与评估模块都不直接服务在线事务，而是作为定期离线任务运行。这样既能利用 Spark 的批处理能力，也便于后续将模型替换为更复杂的特征工程和机器学习模型。

## 4.7 数据库设计
系统数据库分为两类表：业务表和分析结果表。业务表包括用户表、题目表、测试用例表、提交表、判题任务表、判题结果表和分析事件表，用于保证在线业务闭环；分析结果表包括基础汇总表、专题分析表、提交量预测表和题目难度表，用于支撑管理员页面和个人分析页面查询。

在表设计上，提交源码、编译错误、运行错误和消息等字段采用大文本类型，避免长源码和堆栈信息被截断；分析结果表采用面向读取的结构化设计，使前端 API 能够直接按天、按题目或按用户维度取数，减少运行时计算负担。

# 第五章 系统详细设计与实现

## 5.1 用户与权限模块实现
用户模块采用 Spring Security + JWT 的方式实现认证与授权。用户注册时，系统将用户名和经过加密处理的密码写入 `users` 表。用户登录成功后，后端签发 JWT 令牌，并在后续请求中通过请求头携带。系统通过配置管理员用户名集合，为管理员和普通用户分配不同角色。前端在应用启动时通过 `GET /auth/me` 恢复当前会话，并基于角色控制路由访问范围。

在接口实现上，普通用户只能访问与自身相关的数据，例如个人提交详情、个人分析页和个人资料页；管理员则可以访问题目管理、测试用例管理、HDFS 导出、系统分析和预测页面。这样的权限划分使系统在工程实现上保持较低复杂度，同时满足了基本安全边界要求。

## 5.2 题目与测试用例模块实现
题目管理模块支持管理员创建新题、修改题目元数据以及维护测试用例。每个题目包含标题、描述、时间限制和内存限制等字段。测试用例模块支持通过页面直接录入输入内容和标准输出内容，由后端自动为其生成 `.in` 和 `.out` 文件并保存到本地测试数据目录。这样设计的优点在于管理员无需手工准备路径式文件，提高了题目录入效率。

系统在普通用户查看题目时，不返回测试用例信息，以避免标准答案泄露；管理员则可以在题目管理页面中查看测试用例列表并执行覆盖或删除操作。这样的接口行为差异在后端通过身份角色判断实现。

## 5.3 提交与判题模块实现
提交模块在收到用户代码后，会首先校验题目是否存在、用户是否具备访问权限以及提交频率是否超出限制。校验通过后，将代码写入 `submissions` 表，并生成对应的 `judge_tasks` 记录。判题 worker 使用定时轮询方式持续获取待处理任务。该方案虽然不如消息队列形式复杂，但在系统规模较小、部署成本受限的情况下更加简单直接，足以满足本科毕业设计系统的需求。

判题时，系统根据语言类型生成不同的编译和运行命令。以 C/C++ 为例，需要先在容器中执行编译，再运行生成的可执行文件；对于 Java 语言，需要先 `javac` 再 `java` 执行；对于 Python，则直接运行解释器。系统通过 `/usr/bin/time` 统计执行时间和内存占用，并在超时、运行错误和编译错误场景下记录相应错误信息。

输出比对阶段使用统一的归一化比较策略，将 Windows 与 Linux 不同换行符统一处理，并忽略末尾空白和额外空行。若输出不一致，则将预期输出和实际输出的预览片段写入判题结果消息中，便于用户快速定位错误。

## 5.4 判题稳定性与安全机制实现
为提高判题稳定性，系统实现了任务重试、工作目录清理和并发控制等机制。对于执行过程中出现的异常，worker 会记录重试次数，并在达到上限后将任务标记为失败；每次判题完成后，系统会删除工作目录中的源码、编译产物和输出文件，避免磁盘堆积；并发控制则通过固定数量的执行线程和资源限制实现，以防止短时间内大量任务同时触发容器竞争。

在安全方面，除 Docker 容器隔离外，系统还实现了提交限流、防止过大代码与异常输出占用空间等措施。虽然这些机制不能覆盖所有安全场景，但对于教学训练环境已具备较好的防护能力。

## 5.5 分析事件导出与 Hive 聚合实现
在判题结果写入后，系统同步向 `analysis_events` 表插入一条记录，字段包括提交编号、用户编号、题目编号、语言、判题结果、运行时间、内存使用和时间戳。管理员可以通过后台接口触发导出，将尚未导出的事件按日期写入 HDFS。为避免 HDFS 文件被 Hive 误解析，系统在后续分析中为外表配置跳过 header 属性，同时在 Spark 任务中也增加了空值过滤，保证数据链路的鲁棒性。

Hive 聚合部分分为基础聚合与专题聚合。基础聚合生成每日总提交量、题目维度统计和用户维度统计；专题聚合生成语言分布、结果分布、小时活跃度、题目结果分布和用户语言偏好等。所有聚合结果先落到 HDFS，再通过脚本导入 MySQL 对应的结果表。

## 5.6 Spark 提交量预测实现
系统首先实现了基于 Spark 的每日提交量预测。该模块从 Hive 表中读取按天聚合的历史提交数，将日期序号作为特征，提交数作为标签，使用线性回归模型估计未来七天的提交趋势。预测结果写入 `analysis_submission_forecast_daily` 表，并通过管理员预测页面展示。

该实现的重点不在于构建复杂预测模型，而在于打通“大数据读取—模型计算—结果回写—前端展示”的完整链路。作为本科毕业设计的一部分，该模块体现了 Spark 在系统中的预测能力，也为后续引入更复杂模型留下了扩展空间。

## 5.7 Spark 题目难度评估实现
在题目难度评估模块中，系统以某一天的题目判题行为为分析对象，利用 Spark 读取 Hive 中的分析事件数据，并按题目维度聚合出总提交数、通过数、AC 率、平均运行时间、平均内存、平均每用户尝试次数以及 WA、RE、TLE 比例。然后系统基于规则构造难度分数：AC 率越低，难度越高；平均尝试次数越大，难度越高；平均运行时间、运行错误比例和超时比例越高，也会提高难度分数。

根据综合得分，系统将题目映射为 Easy、Medium 和 Hard 三档标签，并将结果写入 `analysis_problem_difficulty_daily` 表。管理员页面进一步展示难度标签分布、最难题目排行、单题难度历史趋势和明细表。该模块虽然属于规则型评估，而非复杂机器学习分类器，但具有可解释性强、实现成本低、对当前数据条件友好等优点，更适合作为系统第一阶段的题目质量评估方案。

## 5.8 前端可视化页面实现
前端采用路由分离方式组织页面。普通用户页面包括题目列表、题目详情、提交结果页、个人资料页和个人分析页；管理员页面包括题目管理页、系统分析页、HDFS 导出页、预测页和题目难度评估页。个人分析页重点展示用户的提交趋势、语言分布、结果分布和最近提交；系统分析页展示全站趋势、语言分布、结果分布、活跃时段和 Top 题目、Top 用户；预测页展示未来提交量趋势；题目难度页展示题目难度标签和历史变化。

前端通过 Axios 调用后端 REST API，并结合 React Query 进行缓存与请求管理。在图表层面，系统使用折线图、柱状图、饼图和表格组合展示信息，使数据分析结果具备较好的可读性和产品化特征。

# 第七章 总结与展望

## 7.1 全文总结
本文围绕“基于大数据计算引擎的在线判题系统的设计与实现”这一课题，完成了一套从在线判题到离线分析再到预测评估的综合系统建设。系统以 Spring Boot 和 MySQL 为业务基础，以 Docker 为安全判题执行环境，以 Hadoop HDFS 作为历史日志存储载体，以 Hive 完成基础聚合与描述性分析，以 Spark 实现提交量预测和题目难度评估，并通过 React 前端完成管理员和普通用户的数据可视化展示。

从实现结果看，系统已经具备完整的业务闭环：普通用户可以完成注册、登录、查看题目、提交代码和查询结果；管理员可以完成题目与测试用例管理、判题日志导出、系统统计查看、预测结果查看以及题目难度评估查看。同时，系统还构建了清晰的数据流路径，即从判题结果生成分析事件，再导出到 HDFS，借助 Hive 与 Spark 完成统计和预测，再将结果回写 MySQL，最终通过后端 API 和前端页面实现展示。这一设计使在线判题系统从单纯的业务执行平台升级为具备数据分析与预测能力的综合平台。

从课题意义看，本文的工作表明，将大数据计算引擎引入在线判题系统具有较高可行性。Hadoop 与 Hive 适合承担历史数据沉淀和离线聚合任务，Spark 适合承担更复杂的预测和评估计算任务，而 Spring Boot 与 MySQL 仍然负责在线业务和结果查询，形成“在线业务与离线分析解耦、结果统一回流”的工程架构。这种设计思路不仅适用于当前的教学训练场景，也具有一定的推广价值。

## 7.2 系统特色与创新点总结
结合全文实现内容，可以将本系统的特色与创新点概括为以下几个方面。

第一，系统在传统在线判题功能基础上，进一步构建了基于 HDFS、Hive 和 Spark 的大数据分析链路，使判题日志不再只是业务副产物，而是成为可持续利用的数据资产。

第二，系统采用 Docker 容器沙箱执行用户代码，结合网络禁用、资源限制、只读挂载等策略，形成较为完整的安全判题执行方案。

第三，系统不仅提供基础统计和描述性分析，还实现了 Spark 提交量预测和题目难度评估两个面向管理决策的离线模块，体现了大数据计算引擎在在线判题场景中的进一步应用。

第四，系统将离线分析和预测结果通过 MySQL 回写并在前端页面中可视化展示，实现了分析结果的产品化落地，使管理员和普通用户都能够直接从页面获得决策信息。

## 7.3 存在的不足
尽管本文实现的系统已经完成了题目要求中的主要目标，但仍存在一些不足。

第一，判题任务目前采用单体应用中的异步 worker 实现，适合中小规模环境，但在更高并发场景下，可能需要进一步拆分判题服务并引入消息队列或分布式调度机制。

第二，Hive 与 Spark 分析任务目前主要依赖手动或脚本触发，自动化调度能力仍有提升空间。若后续采用 Airflow 或更完善的任务编排工具，可以进一步提高离线分析链路的自动化水平。

第三，提交量预测模型目前较为简单，主要用于验证预测链路是否可用，尚未引入更多时间序列特征和评价指标。题目难度评估同样以规则模型为主，虽然解释性强，但在精度和适应性方面仍有提升空间。

第四，前端页面虽然已经具备分析与预测展示能力，但界面美观性、交互深度和移动端适配仍有改进余地。

## 7.4 未来展望
针对上述不足，后续可以从以下几个方向继续完善系统。

第一，进一步扩展 Spark 预测与分析能力。例如引入活跃用户预测、题目热度预测、小时级负载预测，甚至基于用户行为数据构建用户画像和个性化推荐模型。

第二，完善题目难度评估方法。当前的规则型难度评估适合作为第一阶段方案，后续可以结合更多特征，如题目标签、提交语言差异、长期历史趋势和早期提交表现，引入监督学习或聚类方法，提高难度评估的精度与解释能力。

第三，优化系统架构。可以考虑将判题服务从主应用中拆分出来，形成更独立的判题节点体系，并通过消息队列提高任务分发效率；同时增加更细粒度的监控、日志追踪和异常告警能力。
