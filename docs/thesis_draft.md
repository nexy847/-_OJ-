# 基于大数据计算引擎的在线判题系统的设计与实现

## 摘要
随着程序设计课程在高校教学中的普及，如何对学生提交的程序进行高效、准确且安全的自动评测，已经成为教学信息化建设中的一个重要问题。传统的人工批改方式存在批改效率低、反馈周期长、评价标准不统一等问题，难以适应大规模程序设计教学与训练的需求。在线判题系统能够对用户提交的源代码进行自动编译、运行和结果比对，并给出相应的判题结论，因此已成为程序设计教学、竞赛训练和能力评价的重要支撑平台。然而，普通的小型在线判题系统在实际应用中仍然面临若干不足：其一，用户提交的代码具有不确定性和潜在风险，若缺乏有效隔离措施，容易对宿主环境造成安全威胁；其二，传统系统更关注单次判题结果，而对长期积累的提交数据缺乏系统化处理与分析，难以为教学管理和学习过程评估提供有效依据；其三，在功能扩展与系统演进过程中，如果缺少清晰的模块划分和数据流设计，系统往往难以维护。

针对上述问题，本文设计并实现了一套基于大数据计算引擎的在线判题系统。系统以后端 Spring Boot 为核心，采用 MySQL 存储业务数据，使用 Docker 构建隔离式判题沙箱，以实现对用户代码的编译与运行环境隔离；同时引入 Hadoop 生态中的 HDFS 与 Hive，对判题过程中产生的分析事件进行离线存储与批量计算，从而实现用户行为统计、题目难度统计、系统运行情况统计等数据分析功能。在系统实现方面，本文完成了用户注册登录、题目管理、测试用例管理、代码提交、异步判题、结果查询、分析事件导出、Hive 聚合分析以及前端可视化展示等功能模块，构建了一个完整的“在线判题 + 离线分析”闭环。

本文首先从程序设计教学场景出发，对在线判题系统的业务需求和非功能需求进行了分析；其次给出了系统总体架构、模块划分和数据库设计；在此基础上，对判题流程、Docker 沙箱机制、权限控制、异常处理、Hadoop 分析链路以及前端页面实现进行了详细说明。最后，本文通过功能测试、判题结果测试、安全性测试和分析链路测试，验证了系统在小型教学场景下的可行性与有效性。测试结果表明，所设计的系统能够较为稳定地完成多语言代码判题任务，并能够基于历史提交数据生成用户分析页面和管理员分析页面，为程序设计教学提供了良好的平台支持。

关键词：在线判题系统；Docker；Hadoop；Hive；Spring Boot；数据分析

## 第一章 绪论

### 1.1 研究背景
程序设计课程是计算机类专业教学中的核心课程之一，其教学效果不仅影响学生的编程能力培养，也直接关系到后续数据结构、算法设计、操作系统、编译原理等课程的学习质量。在传统教学场景中，程序设计作业的批改往往由教师或助教手动完成，这种方式在学生数量较少时尚可维持，但在班级规模扩大、实践任务增多的情况下，会迅速暴露出效率低、工作量大和反馈滞后的问题。特别是在当前高校课程建设不断强调“过程性评价”和“实践能力导向”的背景下，教师不仅需要知道学生是否完成题目，还希望及时掌握学生在不同知识点上的掌握情况、错误类型分布以及学习活跃度变化。因此，构建一套能够自动判题并支持行为分析的系统，具有明显的现实意义。

在线判题系统作为一种典型的自动评测平台，能够在学生提交代码后自动完成编译、运行和结果比对，并以统一的标准给出判题结果，如编译错误、运行错误、答案错误、时间超限和答案正确等。这种机制能够显著提高教学反馈效率，减轻教师重复批改的负担，并为学生提供及时、客观和可重复的训练环境。近年来，随着云计算、容器化和大数据技术的发展，在线判题系统的实现方式也在不断演进。传统的判题系统多在宿主机中直接调用编译器和运行程序，但由于用户代码不可控，这种方式容易带来安全风险。例如，恶意代码可能尝试访问网络、无限占用资源，甚至借助系统漏洞影响服务器稳定性。因此，如何为用户代码提供安全隔离的运行环境，是在线判题系统设计中的关键问题。

与此同时，在线判题系统在日常运行中会积累大量与教学行为相关的数据，包括用户提交时间、使用语言、判题结果、运行时间、内存消耗以及用户与题目之间的交互频率等。这些数据不仅仅是系统运行日志，也是一种具有价值的教学过程数据。如果能够对这些历史数据进行系统化存储和批量分析，就可以从中提取出学生学习趋势、题目难度特征、用户活跃度分布以及平台运行状态等信息，从而为教师制定教学策略、为学生提供个性化反馈提供数据支撑。然而，许多轻量级 OJ 系统只完成了“判题”这一核心动作，对历史数据分析支持较弱，导致数据价值未能充分发挥。

在这种背景下，将大数据计算引擎引入在线判题系统具有较高研究价值。大数据计算引擎并不意味着系统必须面对互联网级数据规模，而是意味着系统可以采用 HDFS、Hive 等工具对历史数据进行分层存储、批量处理和统计分析，以建立标准化的数据分析链路。这种设计使在线判题系统不仅是一个“答题与判题”平台，也成为一个面向程序设计教学的数据基础平台。因此，研究并实现一套基于大数据计算引擎的在线判题系统，具有较强的工程实践意义和教学应用价值。

### 1.2 研究目的与意义
本文的研究目的在于设计并实现一套面向教学场景的在线判题系统，使其不仅能够完成安全、准确的代码评测，还能够将判题过程中的关键事件沉淀为可分析数据，并借助大数据计算引擎进行离线统计与可视化展示。相较于仅支持单次判题的普通系统，本文系统强调“判题执行”和“教学分析”双重能力：前者保证学生能够完成代码训练与即时反馈，后者保证教师和学生都能从历史数据中获得结构化认知。

从教学层面来看，本系统能够提升程序设计课程的过程性评价能力。学生在完成题目训练时，可以获得即时判题反馈，缩短学习闭环，提高练习效率；教师则可以基于系统统计出的题目通过率、学生提交分布、语言使用情况等信息，判断课程难点和教学薄弱点。这种数据驱动的教学支持，是传统批改模式难以实现的。

从技术层面来看，本文系统将 Docker 判题沙箱与 Hadoop 分析链路结合起来，形成了一个较完整的工程实现案例。Docker 解决了不可信代码执行的问题，使得系统在资源限制、网络隔离、权限控制等方面具备较好的安全性；Hadoop 和 Hive 则解决了历史事件分析的问题，使系统具备批量统计和离线分析能力。这种“在线业务 + 离线分析”的架构模式具有较强的可扩展性，适用于后续继续增加分析指标、个性化推荐和数据挖掘功能。

从个人毕业设计的角度来看，该系统覆盖了后端服务设计、数据库建模、容器化执行、安全控制、分布式存储、离线统计、前端展示等多个知识领域，能够体现较完整的软件工程能力。系统既具备可展示的功能界面，也有可描述的技术难点与实验内容，因此具有较高的论文写作价值。
### 1.3 国内外研究现状
在线判题系统最早主要应用于程序设计竞赛和算法训练领域。随着 ACM/ICPC、OI 和各类编程竞赛的发展，国内外出现了多种知名的 OJ 平台，如 UVA Online Judge、POJ、Codeforces、AtCoder、LeetCode、HDU 等。这些平台在题目组织、竞赛支持、实时排名和自动评测方面积累了较成熟的经验。其共同特点是：以自动编译与结果比对为核心，通过标准输入输出机制对程序正确性进行判定，并以较为统一的方式给出判题结果。

在教学领域，不少高校和培训机构也构建了面向课程实践的在线判题平台。这类系统通常更关注课程作业管理、学生账号体系和教学管理需求，而不是竞赛排名和大规模公开题库。国内部分高校采用自研系统或在开源项目基础上进行二次开发，例如 HUSTOJ、QDUOJ、Judge0 等。这些系统在题目管理、提交管理和基础判题方面功能较为完整，但在教学分析和数据驱动决策方面的支持通常有限。

关于代码执行安全问题，早期系统往往直接在宿主机调用编译器和运行用户程序，配合 chroot、ulimit、seccomp 等系统级手段进行约束。随着容器化技术的发展，Docker 因其部署方便、隔离相对完整、资源限制清晰而逐渐成为在线判题实现中的常用选择。近年来许多研究和工程实践都将 Docker 作为判题沙箱技术基础，通过设置只读文件系统、禁止网络、限制 CPU 和内存等方式来降低不可信代码执行风险。因此，基于 Docker 的判题架构已成为当前较具工程可行性的方案之一。

在数据分析方面，传统 OJ 系统多停留在简单统计层面，例如总提交数、通过数和排行榜等。随着大数据技术在教育信息化中的应用，越来越多研究开始关注程序设计教学中的学习行为分析、知识掌握评估和学习路径建模。Hadoop 生态中的 HDFS 和 Hive 在日志分析、批量统计和历史数据挖掘方面具有成熟的应用基础，适合用于构建教学数据分析平台。然而，将 Hadoop 分析链路与在线判题业务深度结合的毕业设计或教学平台案例相对较少，特别是能够同时覆盖判题、安全隔离、离线分析和前端可视化的完整实现案例更少。因此，本文工作在工程集成和教学场景适配上仍具有一定的实践意义。

### 1.4 本文主要工作
本文围绕“基于大数据计算引擎的在线判题系统”这一主题，完成了以下主要工作：

第一，完成了系统需求分析与总体设计。结合程序设计教学场景，对用户角色、核心业务流程、功能需求和非功能需求进行了梳理，并设计了包括后端服务、MySQL 数据库、Docker 判题沙箱、HDFS 存储、Hive 聚合分析以及前端页面在内的总体架构。

第二，设计并实现了在线判题核心流程。系统支持题目创建、测试用例管理、用户代码提交、判题任务生成、异步判题执行和结果查询等业务流程，并根据编译结果、运行结果和输出比对情况，给出 AC、WA、CE、RE、TLE 等判题结论。

第三，实现了基于 Docker 的安全判题沙箱。系统针对用户提交代码构建独立容器环境，设置网络禁用、进程数限制、内存限制、CPU 限制、只读根文件系统和临时目录等安全策略，以提升判题安全性和稳定性。

第四，构建了基于 Hadoop 的离线分析链路。系统在每次判题结束后生成分析事件，将其导出至 HDFS，以 Hive 进行日统计聚合，并将聚合结果回写 MySQL，为前端分析页面提供查询支持。

第五，完成了前端页面设计与实现。前端基于 React 与 Ant Design 实现了登录注册、题目浏览、提交结果查看、管理员题目管理、测试用例管理以及用户和管理员分析页面。

第六，对系统进行了功能测试和链路验证。通过对不同语言和不同判题结果场景进行测试，验证了判题逻辑、安全机制和分析模块的可用性。

## 第二章 关键技术介绍

### 2.1 Spring Boot 框架
Spring Boot 是在 Spring 生态基础上发展起来的快速开发框架，能够通过自动配置、约定优于配置等机制，降低 Java Web 应用的开发复杂度。对于本系统而言，Spring Boot 主要承担后端核心服务的实现，包括用户认证、题目管理、提交管理、判题调度、分析查询和定时导出等功能。系统利用 Spring MVC 实现 RESTful API，利用 Spring Security 实现认证与授权控制，利用 Spring Data JPA 实现数据库访问，利用定时任务机制实现判题轮询和分析导出调度。

选择 Spring Boot 作为后端框架，主要有以下原因：首先，其生态成熟，适合快速构建结构清晰的业务系统；其次，Spring Security 与 JWT 结合方便，能够较好地实现用户权限管理；再次，JPA 能够简化实体与数据库表之间的映射关系，适合中小型项目开发；最后，Spring Boot 的配置管理与日志机制完善，便于系统部署与维护。对于毕业设计而言，Spring Boot 还具有文档多、社区活跃、上手成本适中的优势。

### 2.2 MySQL 数据库
MySQL 是当前广泛应用的关系型数据库管理系统，具有部署方便、稳定性较好、工具链成熟等特点。在本文系统中，MySQL 主要承担两类数据存储任务：一类是在线业务数据，例如用户、题目、测试用例、提交记录、判题任务和判题结果；另一类是离线分析后的汇总数据，例如按日汇总的全站统计、题目统计和用户统计结果。

采用 MySQL 的优势在于：其一，关系型模型适合描述在线判题系统中实体与实体之间的关系，例如用户与提交、题目与测试用例、提交与判题结果之间的关联；其二，MySQL 对查询型业务支持较好，适合前端页面对题目列表、提交详情和汇总分析结果的访问；其三，系统离线分析后的结果回写至 MySQL 后，可以避免前端直接访问 Hive，提高页面响应速度和使用便利性。因此，MySQL 在本系统中同时扮演“核心业务库”和“分析结果服务库”的角色。

### 2.3 Docker 容器技术
Docker 是一种轻量级容器化技术，能够将应用及其运行环境打包并在隔离环境中执行。与传统虚拟机相比，容器启动速度快、资源占用较低，非常适合在线判题这类“短时间、高频率、需要快速创建执行环境”的业务场景。在本系统中，不同编程语言对应不同的判题镜像，例如 C/C++、Java 和 Python 分别使用各自预构建的镜像，以保证编译器和运行时环境的独立性。

Docker 在本文系统中的主要作用是实现判题沙箱。系统在执行用户代码时，为每次提交创建一个容器，挂载只读测试数据目录和当前提交的工作目录，在容器中执行编译或运行命令，并通过 Docker 参数限制 CPU、内存和进程数，禁止网络访问，并丢弃不必要的系统能力。通过这些限制，可以降低恶意代码、死循环程序和资源滥用对宿主机带来的影响。与此同时，容器的文件系统隔离也使每次判题执行相互独立，提升了系统稳定性。

### 2.4 Hadoop、HDFS 与 Hive
Hadoop 是一套用于分布式存储和分布式计算的开源框架，广泛应用于大规模数据处理场景。虽然本文系统在数据量级上尚未达到互联网级规模，但在设计上引入 Hadoop 生态，主要是为了形成标准化的离线分析链路，为后续扩展更复杂的数据挖掘能力打下基础。

HDFS 是 Hadoop 的分布式文件系统，适合存储大量顺序写入、批量读取的数据。在本系统中，判题完成后产生的分析事件会先存储于 MySQL 的 `analysis_events` 表，然后通过导出任务按日期分区写入 HDFS。例如，某日的事件数据会写入 `/oj/analysis/dt=YYYY-MM-DD/` 目录。这样做的好处是原始数据按天分区、结构清晰，便于后续使用 Hive 进行统计。

Hive 是建立在 Hadoop 之上的数据仓库工具，支持使用类 SQL 语法对 HDFS 上的数据进行查询和聚合。在本系统中，Hive 主要承担离线统计任务，例如计算某日的总提交量、通过量、平均耗时、平均内存，以及题目维度和用户维度的统计结果。聚合结果再写回 MySQL，供前端页面直接调用。采用 Hive 的好处在于：一方面可以将原始事件与在线业务隔离，避免影响主数据库；另一方面能够用接近 SQL 的方式快速定义统计逻辑，降低实现成本。
### 2.5 React、Ant Design 与 ECharts
前端部分采用 React 作为主框架，配合 React Router 构建路由页面，使用 Ant Design 实现表单、表格和布局组件，并使用 ECharts 进行数据可视化展示。React 采用组件化开发模式，便于将登录、题目页、管理页、分析页等模块拆分为独立页面和可复用组件。Ant Design 作为成熟的后台风格 UI 组件库，适合本系统“功能优先、视觉简洁”的前端需求。ECharts 则提供了丰富的折线图、柱状图和饼图配置能力，可用于展示提交趋势、语言分布、判题结果分布和题目通过率等统计信息。

在本文系统中，普通用户页面主要展示题目列表、题目详情、提交详情和个人分析页面；管理员页面则展示题目管理、测试用例管理、系统分析以及 HDFS 导出操作。通过前后端分离架构，前端仅负责页面交互和数据展示，业务逻辑仍集中在后端，保证了系统结构清晰。

### 2.6 JWT 认证机制
JWT，即 JSON Web Token，是一种常见的无状态认证机制。用户登录成功后，后端生成带有用户标识信息的 Token，前端将其保存在本地存储中，后续请求通过 HTTP 头部携带该 Token，后端据此识别当前用户身份。相较于传统基于 Session 的认证方式，JWT 更适合前后端分离架构，且无需服务端维护复杂的会话状态。

在本系统中，普通用户和管理员均通过 JWT 进行认证。管理员身份并不单独存储在数据库角色字段中，而是由后端配置项中的管理员用户名列表进行判断。系统提供了 `GET /auth/me` 接口，用于前端在页面刷新后恢复用户身份和权限信息。通过 Spring Security 的鉴权配置，可以限制只有管理员才能访问题目管理、测试用例管理和系统分析页面，而普通用户只能访问个人数据和题目提交功能。

## 第三章 需求分析

### 3.1 功能需求分析
在线判题系统的功能需求可分为普通用户需求和管理员需求两大类。

对于普通用户而言，系统首先需要提供基础的用户注册和登录功能，使用户能够以个人身份参与系统使用。登录后，普通用户应能够浏览题目列表，查看某道题目的标题、描述、时间限制和内存限制，并在题目详情页面提交自己的代码。提交后，系统需要异步完成判题任务，并允许用户查询当前提交状态以及最终判题结果。此外，考虑到教学场景中的过程反馈需求，普通用户还应能够查看自己的分析页面，包括提交总量、通过率、语言分布、判题结果分布、题目统计以及最近提交记录等内容。为了提高系统的完整性，普通用户还应具备修改个人用户名和密码的能力。

对于管理员而言，系统需要提供更完整的管理能力。管理员应能够创建题目，维护题目元数据，例如标题、描述、时间限制和内存限制；同时，还应能够管理测试用例，包括新增、覆盖和删除测试数据。管理员不仅可以查看所有题目，还应能够通过专门的管理页面查看指定题目的测试用例列表和元数据编辑表单。在数据分析层面，管理员应能够访问系统分析页面，查看全站提交趋势、语言分布、判题结果分布、热门题目和活跃用户等统计结果。此外，管理员还需要具备手动触发分析事件导出到 HDFS 的能力，以配合 Hadoop 离线分析链路运行。

从系统业务流程上看，在线判题核心需求还包括：提交记录入库、判题任务生成、Worker 异步取任务、Docker 容器判题、结果落库、事件记录、HDFS 导出、Hive 聚合以及前端查询展示。各功能模块之间既相互独立，又通过数据流实现联动，因此在需求设计时必须保持清晰的边界与协调关系。

### 3.2 非功能需求分析
除功能需求外，在线判题系统还必须满足若干非功能需求。

第一是安全性要求。由于系统执行的是用户提交的任意源代码，若没有有效隔离，极易带来系统风险。因此，系统必须在代码执行环境中禁用网络、限制资源、控制进程数，并尽量减少容器权限。此外，系统还需要通过 JWT 与权限控制机制，防止非管理员访问受限接口。

第二是稳定性要求。判题系统在运行过程中可能面临编译失败、程序崩溃、无限循环、输出过大、Docker 命令异常等多种情况。系统应能正确区分用户程序错误与系统执行错误，并在必要时支持任务重试、失败标记和工作目录清理，避免单次异常影响后续任务执行。

第三是可维护性要求。作为毕业设计系统，后续仍可能增加新语言、新分析指标和更多管理功能。因此，系统需要具备清晰的模块划分，例如控制器层、服务层、数据访问层、实体层和前端页面层之间职责明确，避免耦合过高。判题逻辑与分析逻辑分离，也是可维护性的重要体现。

第四是可扩展性要求。系统在当前阶段定位为小型教学场景，不要求高并发分布式判题，但在设计上仍应考虑后续扩展可能，例如未来可以引入消息队列、分布式判题节点、更多 Hive 统计表或 Spark 特征工程任务。因此，当前架构应尽量避免把在线判题与离线分析混杂在一起。

第五是可用性要求。前端页面应尽量简洁直观，接口返回结构应稳定，出错信息应可理解，避免因前端交互复杂或错误提示不清导致使用成本过高。

### 3.3 业务流程分析
从业务角度看，本系统的核心流程可以概括为“用户操作驱动在线判题，判题结果沉淀分析事件，离线计算形成数据画像”。

首先，用户通过注册和登录进入系统，在题目列表中选择题目并查看详情。用户在题目页面编写或粘贴源代码，选择相应编程语言后提交。后端收到请求后，将代码、语言、题目编号和用户编号写入 `submissions` 表，并同时创建一条待执行的判题任务写入 `judge_tasks` 表。此时，提交记录状态为 `PENDING`。

随后，系统中的判题 Worker 通过定时轮询机制扫描待处理任务，尝试抢占任务并将其状态置为 `RUNNING`。当任务被成功领取后，系统会根据语言类型选择对应的 Docker 镜像，将源码写入工作目录，必要时在容器中进行编译，再针对题目对应的每个测试用例依次运行程序并比较输出结果。执行完成后，系统生成 `JudgeRunResult`，将其保存到 `judge_results` 表，并更新 `submissions` 中的状态和判题结论。如果执行过程中出现系统级异常，例如容器命令失败或数据库字段过短导致保存失败，则 Worker 会按重试策略重新调度，最终失败时将任务和提交标记为 `FAILED` 与 `ERROR`。

在判题成功写入结果后，系统还会生成一条分析事件写入 `analysis_events` 表。分析事件中包含提交编号、用户编号、题目编号、语言、判题结论、运行时间和内存等信息。这些事件随后由管理员手动触发或通过定时任务批量导出到 HDFS 的日期分区目录中。Hive 通过外表映射 HDFS 数据，对这些事件进行聚合，生成面向管理员和普通用户的数据统计结果，例如每日总提交数、题目通过率、用户提交趋势等。聚合后的结果回写 MySQL，前端页面则通过调用相应 API 展示图表和表格。

### 3.4 可行性分析
从技术角度看，本系统采用的主要技术均具有成熟的开源实现和较好的工程可行性。Spring Boot、MySQL、Docker、React、Hadoop 和 Hive 都具备完善的生态与文档支持，能够满足毕业设计项目的开发需求。系统虽然覆盖模块较多，但每个部分的职责相对明确，适合在有限时间内分步实现。

从经济角度看，系统所使用的软件环境基本均为开源软件，不需要额外采购授权。Docker Desktop 可在本地开发机上运行，Hadoop 集群可部署在若干虚拟机中，整体硬件要求也较低。因此，该项目具有较好的经济可行性。

从应用角度看，程序设计课程中的题目练习、作业管理和过程分析具有真实需求，系统具备明确应用场景。即便在当前规模下仅服务少量课程用户，也能够体现明显的使用价值。因此，该系统具备较好的应用可行性。

## 第四章 系统总体设计

### 4.1 系统设计目标
根据前述需求，本文系统设计时主要围绕以下几个目标展开。第一，保证判题功能完整可用，即用户能够完成题目浏览、代码提交和结果查询的完整闭环。第二，保证代码执行环境安全可控，即不可信代码不能直接影响宿主环境。第三，保证分析链路清晰可扩展，即在线业务数据与离线分析数据分层设计。第四，保证系统结构易维护，即前后端、业务层、数据层和分析层职责边界清晰。第五，保证系统能够为不同用户提供差异化的可视化页面，体现系统在教学场景中的实际价值。

### 4.2 总体架构设计
系统总体上采用前后端分离架构，并将判题执行与离线分析分别置于不同的处理链路中。整体架构可以分为五个层次。

第一层是前端展示层。前端采用 React 构建单页应用，通过路由区分登录页、注册页、题目列表页、题目详情页、提交详情页、个人资料页、用户分析页和管理员页面。前端通过 Axios 调用后端 REST API，并使用 React Query 维护异步请求状态。图表部分使用 ECharts 完成。

第二层是后端业务服务层。后端基于 Spring Boot 构建，主要负责用户认证、题目管理、提交管理、判题任务调度、结果查询、分析查询与 HDFS 导出等功能。后端根据业务模块划分为控制器层、服务层、数据访问层和实体层，各层职责相对明确。
第三层是数据存储层。MySQL 用于存储业务实体数据和汇总统计结果。业务实体包括 `users`、`problems`、`testcases`、`submissions`、`judge_tasks`、`judge_results` 和 `analysis_events`；汇总统计表包括 `analysis_summary_daily`、`analysis_problem_daily` 和 `analysis_user_daily` 等。

第四层是判题执行层。判题执行由后端中的 Worker 异步触发，Worker 从 `judge_tasks` 表中轮询待处理任务，并通过 Docker 调用相应语言镜像执行编译与运行流程。工作目录位于宿主机指定路径，通过挂载方式映射至容器；测试数据目录以只读方式挂载至容器，供程序读取输入并输出结果。

第五层是大数据分析层。后端将分析事件导出到 HDFS 的日期分区目录，Hive 通过外表映射这些原始事件，执行日统计聚合，并将聚合结果回写 MySQL。前端分析页面只需要从 MySQL 查询，不直接访问 Hive，从而保证响应速度与接口统一性。

### 4.3 模块划分设计
系统按职责可以划分为以下模块。

用户模块：负责注册、登录、获取当前用户信息和修改个人资料。该模块与 JWT 认证机制结合，决定用户身份恢复和权限控制逻辑。

题目模块：负责题目创建、题目查询和题目元数据修改。普通用户只能查看题目，管理员可以新增和编辑题目。

测试用例模块：负责测试用例内容写入、覆盖和删除。系统不要求管理员直接维护文件路径，而是由后端根据输入内容和输出内容自动生成 `.in` 与 `.out` 文件并保存路径。

提交模块：负责记录用户提交的代码、语言和题目编号，创建对应的判题任务，并向前端返回提交编号。

判题模块：负责异步领取任务、调用 Docker 执行代码、统计运行指标、对比输出和生成最终判题结果。该模块是系统的核心模块。

分析模块：负责判题事件记录、HDFS 导出、Hive 聚合和 MySQL 汇总表查询。用户分析页和管理员分析页均依赖此模块。

管理模块：负责管理员视角下的题目管理、测试用例管理、数据导出以及全站统计展示。

### 4.4 数据库设计
为了支持系统业务与分析功能，本文设计了多张核心数据表。

`users` 表用于存储用户基础信息，包括用户编号、用户名、密码哈希和创建时间。系统当前将管理员身份通过配置文件中的用户名列表进行识别，因此用户表本身不额外保存角色字段。

`problems` 表用于存储题目元数据，包括编号、标题、描述、时间限制、内存限制和创建时间。题目描述支持较长文本内容，以满足题面展示需要。

`testcases` 表用于存储测试用例的元信息，包括所属题目编号、输入文件路径、输出文件路径和权重。测试用例的实际内容存放于文件系统指定目录下，数据库只记录路径与管理信息。

`submissions` 表用于存储用户提交的源代码及其状态信息，包括用户编号、题目编号、语言、源码、状态、判题结论和时间戳。由于源代码长度可能较大，因此 `code` 字段应使用 `MEDIUMTEXT` 类型而非较小的文本类型。

`judge_tasks` 表用于存储异步判题任务，包括任务状态、重试次数、创建时间、更新时间和下次执行时间。Worker 通过扫描该表实现异步调度。

`judge_results` 表用于存储判题结果，包括提交编号、判题结论、运行时间、内存占用、编译错误、运行错误和提示信息。由于编译错误和运行错误可能包含较长的输出文本，因此相关字段同样应使用 `MEDIUMTEXT` 类型。

`analysis_events` 表用于存储分析事件，是连接在线业务与离线分析的桥梁。每条事件记录一次判题完成后的核心信息，包括用户、题目、语言、判题结论、运行时间、内存和导出状态等。

在离线分析回写层，系统还设计了 `analysis_summary_daily`、`analysis_problem_daily`、`analysis_user_daily` 等表，分别用于存储按日汇总的全站统计、题目维度统计和用户维度统计结果。这些表的数据由 Hive 聚合后导入 MySQL，供前端图表调用。

### 4.5 接口设计原则
系统接口设计遵循 RESTful 风格。普通用户接口主要包括注册登录、当前用户查询、题目列表查询、题目详情查询、提交代码、查询提交状态、查询判题结果和个人分析接口；管理员接口主要包括题目创建、题目更新、测试用例新增与维护、HDFS 导出和系统分析接口。

接口设计强调三点。首先，路径表达业务含义，例如 `/problems/{id}`、`/submissions/{id}`、`/analytics/user/overview`。其次，权限控制应由后端统一保证，前端仅作为页面层进行隐藏或导航判断，而不作为安全边界。第三，返回结构应尽量稳定，使前端页面在开发和维护过程中能够稳定依赖。

## 第五章 系统详细设计与实现

### 5.1 用户与认证模块实现
用户模块首先支持注册功能。新用户通过提交用户名和密码即可在系统中创建账号。后端在创建用户时使用密码编码器对密码进行加密存储，数据库中仅保存密码哈希，而不保存明文密码。为兼容早期数据，系统在登录逻辑中还保留了对旧哈希方式的兼容处理，若检测到用户使用旧哈希格式，则在登录成功后自动迁移为新的加密形式。

登录模块采用 JWT 实现无状态认证。用户登录成功后，后端生成包含用户名信息的 Token，并返回给前端。前端将 Token 保存在本地存储中，并在后续请求中通过请求头自动携带。为支持页面刷新后的身份恢复，系统额外提供 `GET /auth/me` 接口，用于根据当前 Token 获取用户编号、用户名和角色信息。前端在应用启动时先调用该接口恢复会话，再决定渲染普通用户页面还是管理员页面。

在权限控制方面，系统利用 Spring Security 对接口进行约束。普通用户可访问自身信息、题目浏览、提交判题和个人分析相关接口；管理员则可额外访问题目管理、测试用例管理和系统分析接口。当前系统将管理员身份配置在后端配置文件的管理员用户名列表中，因此在鉴权时只需判断当前用户名是否属于管理员集合即可。

此外，本文还实现了个人资料修改功能。普通用户可以在前端的 Profile 页面修改自己的用户名与密码。为了保证安全性，修改资料时必须输入当前密码，后端先验证当前密码是否正确，再更新用户名和密码哈希。由于 JWT 中保存的是旧用户名，因此用户修改资料后前端会主动清除当前登录态，要求用户重新登录，以避免 Token 与数据库中的身份信息不一致。

### 5.2 题目与测试用例模块实现
题目模块支持管理员创建题目、查看题目和修改题目元数据。创建题目时，管理员需要输入标题、题目描述、时间限制和内存限制。系统同时支持两种测试用例录入方式：一种是在创建题目时直接填写若干测试用例内容；另一种是在题目创建完成后进入管理页面逐条新增测试用例。这样既兼顾了快速录入的便利性，也保留了后续维护的灵活性。

在测试用例管理方面，系统不要求管理员手动指定测试文件路径。管理员在页面中输入的是测试用例的输入内容和标准输出内容，后端通过 `TestcaseStorageService` 将内容写入指定目录下的 `.in` 与 `.out` 文件，并返回对应路径写入数据库。对于覆盖测试用例功能，系统根据测试用例已有路径直接覆盖文件内容，从而保持数据库路径不变，降低维护复杂度。删除测试用例时，系统会同时删除数据库记录和对应的文件。

普通用户在访问题目详情页时只能看到题目元数据和题面内容，不能直接访问测试用例信息；管理员则可在题目管理页面中查看测试用例列表、输入文件路径、输出文件路径和权重等管理信息。为了使管理页面职责更明确，本文后续还将题目元数据编辑表单纳入 `Manage Problem` 页面中，使管理员在同一页面即可完成题目元数据和测试用例的整体维护。

### 5.3 提交与异步判题实现
当普通用户在题目详情页提交代码时，前端会将题目编号、语言和源码通过 `POST /submissions` 接口发送给后端。后端首先根据当前登录用户身份设置 `userId`，对代码大小进行校验，防止超长代码直接占用过多资源；随后将该提交写入 `submissions` 表，并设置状态为 `PENDING`。与此同时，系统会为该提交创建一条 `judge_tasks` 记录，用于后续异步调度。

异步判题由 `JudgeWorker` 负责。Worker 使用 Spring 的定时任务机制，按固定间隔扫描 `judge_tasks` 表中状态为 `PENDING` 的任务，并结合 Semaphore 和线程池控制并发数量。每当 Worker 成功抢占一个任务时，会将其状态改为 `RUNNING`，再将具体判题逻辑交给 `JudgeService` 执行。这样设计的优势在于：提交接口只负责快速落库和创建任务，不阻塞前端请求；判题流程则在后台异步进行，提高系统响应能力。
如果判题过程中发生系统级异常，例如 Docker 调用异常、结果写库失败等，Worker 会根据配置的最大重试次数和延迟时间决定是否重试。当重试次数耗尽后，系统会将对应提交标记为 `FAILED`、判题结论标记为 `ERROR`，同时将任务状态标记为 `FAILED`。这与普通的编译错误或运行错误不同，后者属于用户程序错误，正常情况下应写入 `judge_results` 并将提交状态标记为 `DONE`。

### 5.4 Docker 沙箱判题实现
Docker 判题模块是本文系统的核心技术实现之一。系统根据提交语言选择对应的镜像，例如 C/C++ 使用 `oj-gcc`，Java 使用 `oj-java`，Python 使用 `oj-python`。在执行判题时，后端先将源码写入宿主机的工作目录，再通过挂载方式将该目录映射到容器中的 `/workspace`；同时，将测试数据目录以只读方式挂载到容器中的 `/testcases`。这样，用户代码只能在容器中访问工作目录和测试数据目录，无法直接访问宿主机其它文件。

在资源限制方面，系统使用 Docker 参数对容器进行多重约束。首先通过 `--network none` 禁止容器访问外网，避免用户代码发起网络请求；其次通过 `--cpus`、`--memory` 和 `--pids-limit` 对 CPU、内存和进程数进行限制，防止恶意程序或死循环耗尽宿主资源；再次，通过 `--read-only` 将容器根文件系统设置为只读，并使用 `tmpfs` 映射 `/tmp` 和 `/var/tmp`，减少用户对容器内部环境的破坏；此外，通过 `--security-opt no-new-privileges` 与 `--cap-drop ALL` 剥离不必要的系统能力，降低权限提升风险。

对于需要编译的语言，系统先在容器中执行编译命令，如 `g++` 或 `javac`。若编译失败，则返回 `CE`，并将编译器输出保存到 `compileError` 字段。对于运行过程，系统使用 `/usr/bin/time` 获取程序执行时间和最大内存占用，并根据题目时间限制设置容器命令超时时间。程序运行结束后，系统读取输出文件，与标准输出文件使用归一化对比算法进行比较。所谓归一化比较，即忽略末尾多余空格、统一换行格式、忽略文件末尾空行等，以减少无关格式差异对判题结果的影响。

### 5.5 判题结果与异常处理实现
系统定义了多种判题结论，包括 `AC`、`WA`、`CE`、`RE`、`TLE`、`MLE`、`ERROR` 等。其中，`AC` 表示输出与标准答案一致；`WA` 表示程序运行成功但输出不符合预期；`CE` 表示编译失败；`RE` 表示运行时异常，例如除零错误、段错误或解释器报错；`TLE` 表示超出时间限制；`ERROR` 则表示判题流程本身发生系统异常，例如题目不存在、测试数据缺失、容器执行失败或结果写入失败。

在实现过程中，系统除了处理用户程序错误外，还需要处理工程层面的异常。例如，曾经在实际测试中出现源代码字段使用 `tinytext` 导致源码过长无法写入数据库的问题，也出现过 `judge_results.runtime_error` 字段过短，无法保存 Python Traceback，从而导致整个判题事务回滚的问题。这类问题说明，在线判题系统不仅要关注业务流程正确性，还必须注意数据库字段容量设计。为避免类似错误，系统后续将 `submissions.code`、`judge_results.compile_error`、`judge_results.runtime_error` 和 `judge_results.message` 等字段统一调整为 `MEDIUMTEXT` 类型，从而更适合存储长源码和错误输出。

对于前端而言，提交详情页会定时轮询提交状态。当提交仍处于 `PENDING` 或 `RUNNING` 时，页面提示“Judging in progress”；当状态为 `DONE` 时，前端进一步查询 `judge_results`，展示时间、内存、提示信息、编译错误或运行错误文本。若任务最终进入 `FAILED` 状态，则说明系统级错误导致未生成正式判题结果，需要结合后端日志进一步定位。

### 5.6 分析事件与 Hadoop 分析链路实现
为了支持教学数据分析，系统在每次判题完成后，会生成一条分析事件记录。该事件包含提交编号、用户编号、题目编号、语言、判题结论、运行时间、内存占用和创建时间等信息，并写入 `analysis_events` 表。分析事件是系统在线业务向离线分析过渡的关键媒介。

在导出环节，系统通过后台服务批量扫描尚未导出的分析事件，并将其写入 HDFS 的日期分区目录。例如，2026-03-17 的事件会导出到 `/oj/analysis/dt=2026-03-17/`。HDFS 中的事件文件以 CSV 形式存储，便于 Hive 外表直接映射。管理员可以通过前端按钮调用导出接口，也可通过后端定时任务自动执行导出。

在离线分析环节，Hive 使用外表读取 HDFS 中的原始事件数据，并按天执行聚合。当前系统聚合出三类核心统计结果：一是全站每日统计，包括每日总提交数、通过数、平均耗时和平均内存；二是题目维度每日统计，包括每道题在某日的提交数和通过率；三是用户维度每日统计，包括每位用户在某日的提交数和通过情况。聚合结果最终回写 MySQL 的 `analysis_summary_daily`、`analysis_problem_daily` 和 `analysis_user_daily` 表。

这种设计的优势在于：在线判题服务无需承担大规模统计计算压力，业务逻辑与分析逻辑相互解耦；原始事件可以长期保存在 HDFS 中，便于后续增加新的统计指标；前端分析页面又只需要查询 MySQL，避免直接依赖 Hive 查询带来的高延迟和复杂性。对于毕业设计而言，这种链路能够较好体现大数据计算引擎在系统中的实际价值。

### 5.7 用户分析页面与管理员分析页面实现
在前端实现中，系统为普通用户和管理员分别设计了不同的数据分析页面。普通用户分析页面侧重于展示个人画像，包括总提交数、通过率、近 30 天提交趋势、语言分布、判题结果分布、题目维度统计和最近提交记录等。这些数据部分来自 `analysis_events` 的实时查询，部分可来自聚合后数据表，具体取决于数据粒度和接口设计。通过这些可视化内容，学生可以更直观地了解自己的练习频率、通过情况和常用语言。

管理员分析页面则强调全站视角。页面中展示了某日总提交量、总通过量、活跃用户数、平均耗时、平均内存，以及近若干天的趋势图、语言分布、判题结果分布、热门题目排行和活跃用户排行等。这些数据主要依赖 Hive 聚合后的 MySQL 汇总表，以保证查询效率和结果稳定性。通过管理员分析页面，教师或系统管理者可以快速了解系统运行情况和教学使用状态。

在页面实现上，前端采用 React Query 管理请求状态，避免手工维护加载和缓存逻辑；图表统一由 ECharts 绘制，表格则使用 Ant Design Table 组件实现。整个前端界面不追求复杂动画，而是强调路由清晰、操作直接、信息可读和错误提示明确，以符合后台管理系统和教学平台的实际使用需求。

## 第六章 系统测试与结果分析

### 6.1 测试环境
本文系统的测试环境包括：操作系统为 Windows，后端运行于 JDK 17 和 Spring Boot 环境，数据库采用 MySQL 8.x，容器运行环境为 Docker Desktop，前端采用 Node.js、Vite 和 React。大数据分析环境部署于三台虚拟机构成的 Hadoop 3.3.4 集群，其中一台承担 NameNode、ResourceManager 与 Hive 服务，其余节点承担 DataNode 和 NodeManager 功能。通过该环境，可以完成从在线判题到 HDFS 导出，再到 Hive 聚合和前端分析展示的完整链路验证。

### 6.2 功能测试
功能测试主要验证系统各核心模块是否可正常使用。首先，对注册登录功能进行测试，结果表明用户能够成功注册新账号、登录系统并通过 JWT 维持会话。其次，对题目模块进行测试，管理员可以创建题目、修改题目元数据，并在管理页面维护测试用例；普通用户则可以浏览题目并查看题面内容。

在提交与判题功能测试中，系统对 C/C++、Java 和 Python 三类语言进行了验证。测试内容覆盖编译成功、编译失败、运行错误、答案错误和答案正确等情况。实际测试表明，系统能够正确区分不同判题结论，并在前端提交详情页中展示对应结果和错误信息。当提交成功并生成结果时，`submissions` 状态被置为 `DONE`，而 `judge_results` 表中会生成对应记录。
在用户分析和管理员分析功能测试中，系统能够根据已有数据展示折线图、饼图和表格。当数据为空时，页面能够正常展示空状态而不报错；当数据存在时，页面能够正确渲染汇总卡片和图表内容。这表明前后端分析接口对接基本稳定。

### 6.3 判题正确性测试
判题正确性测试重点关注不同判题结果是否与预期一致。对于正确答案样例，系统应返回 `AC`；对于输出错误的程序，应返回 `WA`；对于编译失败的源代码，应返回 `CE`；对于运行中抛出异常或程序崩溃的情况，应返回 `RE`；对于超时场景，应返回 `TLE`。在实际测试中，系统针对每种场景均构造了对应样例，并能给出正确结果。

例如，在括号匹配题中，提交一段语法正确但逻辑错误的程序时，系统会读取其输出并与标准答案进行归一化比较，最终返回 `WA`；提交一段将 Python 代码错误地标记为 C++ 语言的记录时，系统会调用 C++ 编译器并返回 `CE`；提交会引用未定义变量的 Python 程序时，系统会在运行阶段得到解释器 Traceback，并应返回 `RE`。这些结果都符合 OJ 系统的常规行为。

在测试过程中，也发现了一些工程实现问题。例如，当源码字段或错误输出字段设计过短时，数据库写入会失败，从而导致判题事务回滚。这类问题在修正数据库字段类型后得到解决，说明系统在业务逻辑层面可行，但工程实现中必须充分考虑文本字段长度与异常输出规模。

### 6.4 安全性与稳定性测试
安全性测试主要从三个方面展开。第一，验证网络禁用。通过在用户程序中尝试访问外部网络资源，系统容器因 `--network none` 设置而无法访问互联网，从而保证了宿主环境与外部资源安全。第二，验证资源限制。通过构造死循环程序和高内存占用程序，系统能够在达到阈值时终止执行，避免宿主机资源被长期占用。第三，验证权限控制。普通用户访问管理员接口时会被拒绝，而管理员则可正常执行管理操作。

稳定性测试主要考察异步任务重试与失败恢复机制。通过构造容器运行异常、数据库字段过短、测试数据缺失等情况，可以观察到系统会在一定次数内重试任务，最终无法恢复时将任务标记为 `FAILED`。这种机制虽然不能替代根因修复，但能够防止异常任务无限循环执行，有助于保障整体稳定性。

### 6.5 分析链路测试
分析链路测试验证从 `analysis_events` 到 HDFS、再到 Hive 和 MySQL 的完整闭环是否可用。首先，在判题完成后检查 `analysis_events` 表，确认系统能够正确记录分析事件。其次，通过导出接口或脚本将事件导出至 HDFS，并在集群上使用 `hdfs dfs -cat` 查看导出文件内容，验证 CSV 格式与字段是否正确。之后，在 Hive 中执行 `MSCK REPAIR TABLE` 与聚合 SQL，确认 Hive 能够读取分区数据并产出目标统计结果。最后，将聚合结果回写 MySQL 并调用前端分析页面进行展示，若页面图表数据正常，则说明分析链路整体可用。

测试结果表明，系统已经具备“在线判题—事件记录—HDFS 导出—Hive 聚合—MySQL 回写—前端展示”的完整分析流程。对于教学场景下的日统计和用户/题目维度统计，该设计具有较好的可操作性。

### 6.6 测试结果分析
综合功能测试、判题测试和分析链路测试，本文系统在小型教学场景下已经能够实现预期目标。系统可以稳定完成题目管理、用户提交、多语言判题和数据统计展示等任务，前后端交互逻辑较为清晰。Docker 沙箱机制有效提升了执行安全性，Hadoop 分析链路也体现了系统在数据分析方面的扩展能力。

当然，系统仍存在一些局限。例如，当前判题模式更适合小规模使用场景，尚未引入消息队列和分布式判题节点；部分数据表字段长度问题是在测试中逐步暴露和修正的，说明数据库设计仍需进一步严谨；分析指标目前主要是描述性统计，尚未扩展到更深层次的特征工程、聚类分析或预测分析。但就毕业设计目标而言，系统已基本满足设计与实现要求。

## 第七章 总结与展望

### 7.1 全文总结
本文围绕“基于大数据计算引擎的在线判题系统的设计与实现”这一主题，完成了一套面向教学场景的在线判题系统的设计、开发与验证工作。系统采用 Spring Boot 构建后端服务，使用 MySQL 管理业务数据和汇总数据，利用 Docker 实现用户代码的隔离执行，并借助 HDFS 与 Hive 构建离线分析链路，最终通过 React 前端页面完成功能展示和数据可视化。

在功能实现方面，系统已支持用户注册登录、题目浏览、代码提交、异步判题、结果查询、题目管理、测试用例管理、个人资料修改、用户分析页面和管理员分析页面等核心功能。系统实现了从业务提交到判题执行、再到结果展示的完整闭环，能够满足小型程序设计教学场景的基本需求。

在技术实现方面，本文重点完成了 Docker 沙箱判题与 Hadoop 分析链路的工程集成。Docker 通过网络隔离、资源限制和能力裁剪等方式，有效降低了不可信代码执行对宿主环境的影响；Hadoop 与 Hive 则为系统提供了历史事件批量分析能力，使在线判题系统具备教学数据分析功能。通过这两类技术的结合，系统兼顾了在线业务的即时性和离线分析的可扩展性。

在测试验证方面，系统对不同语言和不同判题结果场景进行了较全面测试，验证了判题逻辑和前后端功能的正确性；同时也完成了 HDFS 导出和 Hive 聚合的链路测试，证明系统具备良好的工程可用性。通过这些测试，本文所设计实现的系统基本达到了毕业设计预期目标。

### 7.2 存在的不足
尽管系统已经实现了核心功能，但仍存在一些不足。首先，当前系统定位于小型教学场景，判题任务通过数据库表和定时轮询方式调度，尚未引入消息队列，因此在高并发场景下的吞吐能力有限。其次，管理员身份目前通过配置项中的用户名列表识别，而不是标准化的角色表设计，这在更复杂的权限管理场景下扩展性一般。

第三，系统的数据分析能力目前主要停留在描述性分析层面，例如提交趋势、通过率、语言分布和活跃用户等统计指标，尚未进一步深入到基于用户行为的画像建模、知识点掌握分析或提交趋势预测等更高级的分析任务。第四，虽然系统已经采用 Docker 沙箱进行隔离，但对于恶意代码的极端防护仍然可以继续加强，例如结合 seccomp 配置、更严格的容器镜像最小化设计等。第五，论文中尚未引入更系统的性能压力测试，例如在多用户并发提交下的判题吞吐对比和资源占用统计，这些都是未来可继续补充的内容。

### 7.3 未来展望
未来可以从以下几个方向继续扩展本系统。第一，在判题架构方面，可以引入消息队列和多判题节点部署，将当前单机异步 Worker 模式扩展为分布式判题架构，从而提高并发处理能力。第二，在权限管理方面，可以引入标准化角色表与 RBAC 权限模型，使管理员、教师、助教和学生等不同角色具有更细粒度的权限划分。

第三，在数据分析方面，可以在现有 Hive 描述性分析基础上引入 Spark 或 PySpark，对 `analysis_events` 进行更深层的数据挖掘。例如，可以基于用户提交行为构建学习画像，基于题目提交分布估计题目难度，或基于历史数据预测活跃用户趋势。这将使系统更符合“基于大数据计算引擎”的长期发展方向。第四，在前端展示方面，可以进一步优化数据可视化布局和图表交互，使用户分析页和管理员分析页更适合教学实际使用。第五，在应用场景方面，可以考虑增加竞赛模式、排行榜、班级管理和作业管理等功能，使系统从个人训练平台进一步演进为课程综合实践平台。
